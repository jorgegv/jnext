## Task 5: Investigate new beast issue

- In beast.nex, about the 20% of the bottom scanlines are scrolling in a parallax fashion: the upper lines are scrolling slowly, with a pace similar to the running character; and going down, scanlines are scrolling faster, so ha the bottom one scrolls much faster. This gives an impression that the part of the screen nearer to the observer scrolls faster, creating an amazing perspective effect.
- CSpect simulates this correctly.
- This cannot be seen in JSpect.
- Neither in ZesarUX

**DONE 2026-04-26.** Source-code analysis of `Beast/copper.asm` +
`Beast/scroll.asm` confirmed the cause: Copper writes
`COP_LAYER2_XOFF` (= NR 0x16, Layer 2 X-scroll) at scanlines
163/165/169/173/179, each `Grass1`..`Grass5` MOVE-target patched per
frame to a different value (8 strips, speeds 1..7) via
`ScrollGrass`. jnext consumed the live (last-write-wins) value at
frame end → entire L2 surface scrolled uniformly. Fix: per-scanline
replay for NR 0x16/0x17/0x71 mirroring the per-scanline palette
pattern (sky-gradient feature from earlier this session). Commit
`f448b4f`. Reviewer APPROVE-WITH-NITS, both nits addressed
(`overflow_warned_` now logs via `Log::video()->warn` matching
PaletteManager; audit doc marks NR 0x16/0x17/0x71 as DONE).
Tests 3336/3336/0/0 (+10 from G10), regression 34/0/0. User confirmed
visually: "it works perfectly".
