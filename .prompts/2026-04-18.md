## Task 12: NextZXOS firmware stall upstream of RESET_SOFT (root cause FIXED 2026-04-18, follow-on open)

**Root cause found and fixed 2026-04-18 EOD.** The stall was caused by an MMU aliasing bug in Next mode:

- In Next mode with `rom_in_sram_=true` (Task 11 Branch 2), `Mmu::map_128k_bank(port_7ffd=0)` set slots 6/7 to SRAM pages 0/1 via raw `bank*2`. But SRAM pages 0/1 are where ROM-in-SRAM stores the Spectrum ROM, so slot 6/7 writes aliased the ROM area.
- The firmware's FATFS struct lives at RAM 0xCA94-0xCCC2 (slot 6). When `display_bootscreen()` called `fwRead(0x0000, 0x2800)`, the boot-screen data writes at 0x0000-0x2800 (routed via config_mode to SRAM page 0 when NR 0x04 = 0x00) **overwrote FATFS** at the same physical location (SRAM page 0, offsets 0xA94-0xCC2).
- Corrupted FATFS → `get_fat()` returned garbage → `f_read` aborted → `display_error("Error reading TBBLUE.FW data!")`.

**VHDL citation:** zxnext.vhd:2964 — `mmu_A21_A13 <= ("0001" + ('0' & mem_active_page(7:5))) & mem_active_page(4:0)`. The +0x20 shift puts port_7ffd bank 0 on SRAM page 0x20 (RAMPAGE_RAMSPECCY base), not page 0.

**Fix (2 hunks):**
- `src/memory/mmu.cpp` `map_128k_bank`: for `rom_in_sram_=true`, base = 0x20; else legacy 0.
- `src/core/emulator.cpp::init()`: after `set_rom_in_sram(true)`, call `mmu_.map_128k_bank(0)` to refresh slots from their RESET_PAGES seed (tbblue.fw never writes port_7FFD during its prologue).

**Empirical result:** firmware now reaches `load_roms()` and displays "Loading ROM: enNextZX.rom... OK!". The Task 12 stall is GONE.

**Follow-on (open):** NextZXOS still does not boot all the way through. Progress beyond load_roms → RESET_SOFT → autoexec is blocked by a different bug. Next session starts fresh there.

**Not revisited by this fix:** `nex_loader.cpp` and `set_page()` callers still use raw `bank*2` (no +0x20 shift). Those paths work today for NEX files because they avoid bank 0, but they are technically wrong under the same VHDL rule — track as a latent bug, fix only if it manifests.
