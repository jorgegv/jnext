### Confirmed real bugs surfaced during Task 5 Step 2 (2026-04-14)

Each item below was empirically verified during Step 2 test-fix work. Tests and VHDL agree; the emulator is wrong. Each has a fix sketch but NONE have been applied (user directive: emulator fixes go to Task 3, not Step 2). Addresses and citations below refer to the repo state at commit `f0b5fa0`.

1. **I2C false STOP detection** — `src/peripheral/i2c.cpp:101`. `detect_start_stop()` is called unconditionally from `write_scl()` but uses `prev_sda_` which was last updated by `write_sda()`. On the first bit of every `i2c_send_byte`, the sequence `write_sda(1)` (prev_sda_←0, sda_out_←1) followed by `write_scl(1)` causes the STOP check `scl==1 && prev_sda_==0 && sda_out_==1` to fire, resetting state to IDLE before the slave can sample. Per I2C spec, START/STOP are SDA transitions — detection should only run from `write_sda`, never from `write_scl`. Fix: remove the `detect_start_stop()` call at line 101. Collapses all 9 UART/I2C baseline failures (I2C-P03, I2C-P05a/b, RTC-01/02/04/05/06/07).

2. **AY envelope hold-at-is_bot off-by-one** — `src/audio/ay_chip.cpp` envelope state machine. Shapes 0 (`\___`) and 0x09 (same pattern with continue bit) output YM `vol_table_ym_[1] = 0x01` instead of `vol_table_ym_[0] = 0x00`. Per VHDL `ym2149.vhd` L411-462, the down+hold branch when alt=0 should latch hold on `is_bot` (env_vol==0), not `is_bot_p1` (env_vol==1). Likely a one-step-too-early hold trigger. Failing tests: AY-110, AY-113.

3. **Turbosound default panning and pan=01 routing** — audit flagged TS-10 ("Reset: all panning to both L+R") as a real emulator bug; Step 2 empirical run also shows TS-42 ("Pan 01: R only, L silenced") broken with `L=0 R=0`. Both point to the Turbosound/YM2149 mixer not initializing panning to `0b11` (both channels) at reset AND the pan register 01 decode being wrong. Likely a single missing init in the Turbosound class reset path plus a decode off-by-one.

4. **ULA frame completion timing** — `test/ula/ula_test.cpp` S13.14 "Frame complete after full T-states" shows `frame_done=0 after 69888 T-states`. S13.01 and S13.02 confirm the emulator's `hc_max()*vc_max()/2 = 69888` (matches VHDL), and S13.03 now asserts this honestly. The bug is that `frame_done` doesn't flip when the T-state counter reaches the frame boundary. Investigate the T-state accumulation path in the ULA/video-timing class.

5. **MMU reset slot clobber** — `Mmu::reset()` sets `slots_[0]=0x00, slots_[1]=0x01` but VHDL `zxnext.vhd:4611-4612` explicitly assigns `MMU0 <= X"FF"; MMU1 <= X"FF"` on reset (0xFF = ROM sentinel). Fix: change the two reset assignments. Risk: boot-path code may assume concrete bank numbers at startup; if so, fix the boot code to respect the ROM sentinel. Failing tests: RST-01, RST-02 in `mmu_test.cpp`. This was originally pre-approved as a cheap Step 2 emulator fix but deferred per the "emulator bugs → Task 3" rule.

6. **DMA zero-length transfer** — `test/dma/dma_test.cpp` tests 9.7 ("Block length = 0: no bytes transferred") and 14.4 ("ZXN: block_len=0 transfers 0 bytes") both show the DMA still transfers 1 byte when `block_len==0`. Per Zilog DMA docs and the VHDL `dma.vhd` state machine, `block_len==0` should transfer zero bytes (not wrap to 65536, not transfer one). Likely a missing zero-check in the transfer loop or an off-by-one in the length decrement.

7. **NextREG reset-defaults tests rewrite as integration tests** — 14 RST-xx failures in `test/nextreg/nextreg_test.cpp` are a test-design issue, not an emulator bug. The bare `NextReg` class doesn't own the VHDL reset defaults (NR 0x12=0x08, 0x14=0xE3, 0x42=0x07, 0x4A=0xE3, 0x4B=0xE3, 0x50-0x57 MMU mirrors, 0x82-0x89 port enables). These defaults live in subsystem classes (Ula, Layer2, Mmu, port dispatcher) that hook handlers on NextReg. When the test constructs a bare `NextReg`, no subsystems are attached, so reads return zero. The real emulator is not broken — in a full `ZxNext` boot, port 0x253B reads return the correct VHDL values. **Fix direction**: rewrite these 14 tests as full-machine integration tests (same pattern as demo/regression tests): construct `ZxNext`, reset, read via port dispatcher, check against VHDL defaults. Do NOT duplicate state by moving defaults into `NextReg::reset()` — that would blur ownership. See `memory/project_nextreg_integration_test.md` for the full rationale.

8. **Tilemap control-bit swap** — from original Task 4 audit (not re-verified in Step 2 runs). `Tilemap::set_control` has bit 3 (text mode) and bit 5 (strip flags) swapped relative to VHDL `tilemap.vhd`. Several tests in `test/tilemap/tilemap_test.cpp` are currently "anti-tests" that document the bug as expected behaviour (TM-CB5 etc.). Fix: correct the bit mapping in the C++ class AND delete the anti-tests, replacing them with VHDL-correct assertions.
