### Open items

7. **NextREG reset-defaults tests rewrite as integration tests** — 14 RST-xx failures in `test/nextreg/nextreg_test.cpp` are a test-design issue, not an emulator bug. The bare `NextReg` class doesn't own the VHDL reset defaults (NR 0x12=0x08, 0x14=0xE3, 0x42=0x07, 0x4A=0xE3, 0x4B=0xE3, 0x50-0x57 MMU mirrors, 0x82-0x89 port enables). These defaults live in subsystem classes (Ula, Layer2, Mmu, port dispatcher) that hook handlers on NextReg. When the test constructs a bare `NextReg`, no subsystems are attached, so reads return zero. The real emulator is not broken — in a full `ZxNext` boot, port 0x253B reads return the correct VHDL values. **Fix direction**: rewrite these 14 tests as full-machine integration tests (same pattern as demo/regression tests): construct `ZxNext`, reset, read via port dispatcher, check against VHDL defaults. Do NOT duplicate state by moving defaults into `NextReg::reset()` — that would blur ownership. See `memory/project_nextreg_integration_test.md` for the full rationale.

27/28. **SPI MISO pipeline delay — PARKED** — VHDL `spi_master.vhd:162-166` latches `miso_dat` one cycle after `state_last`, so the first read after select returns the *previous* exchange result. C++ `SpiMaster::read_data` returns the current exchange result. Implementing pipelined reads requires a `SpiDevice` receive/send split-stream refactor that risks breaking the SdCard boot path. SX-02/SX-03 mutual exclusivity resolved in test plan but implementation deferred. Failing tests: SX-03, ML-05 (+ SS-10 from write_cs edge case).
