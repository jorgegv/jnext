### Confirmed real bugs surfaced during Task 5 Step 2 (2026-04-14)

Each item below was empirically verified during Step 2 test-fix work. Tests and VHDL agree; the emulator is wrong. Each has a fix sketch but NONE have been applied (user directive: emulator fixes go to Task 3, not Step 2). Addresses and citations below refer to the repo state at commit `f0b5fa0`.

1. ~~**I2C false STOP detection**~~ — **DONE 2026-04-15** (merge commit on main). Removed `detect_start_stop()` from `write_scl()`; unblocked 8 rows (I2C-P03, I2C-P05a, I2C-P05b, RTC-01, RTC-02, RTC-04, RTC-05, RTC-06). RTC-07 still fails — separate DS1307 day-of-week BCD issue, tracked separately.

   Original diagnosis: **I2C false STOP detection** — `src/peripheral/i2c.cpp:101`. `detect_start_stop()` is called unconditionally from `write_scl()` but uses `prev_sda_` which was last updated by `write_sda()`. On the first bit of every `i2c_send_byte`, the sequence `write_sda(1)` (prev_sda_←0, sda_out_←1) followed by `write_scl(1)` causes the STOP check `scl==1 && prev_sda_==0 && sda_out_==1` to fire, resetting state to IDLE before the slave can sample. Per I2C spec, START/STOP are SDA transitions — detection should only run from `write_sda`, never from `write_scl`. Fix: remove the `detect_start_stop()` call at line 101. Collapses all 9 UART/I2C baseline failures (I2C-P03, I2C-P05a/b, RTC-01/02/04/05/06/07).

2. **AY envelope hold-at-is_bot off-by-one — wider than shape 0/9** — `src/audio/ay_chip.cpp` envelope state machine. Per VHDL `ym2149.vhd` L411-462 the down+hold branch when alt=0 should latch hold on `is_bot` (env_vol==0), not `is_bot_p1` (env_vol==1) — a one-step-too-early hold trigger. **Scope widened by Wave 1 Audio rewrite (2026-04-15)**: failing tests are AY-81, AY-102, AY-110, AY-122 (not just the originally-identified shape 0/9). AY-113 previously thought wrong is actually VHDL-correct at YM=0x01 (plan wording "hold at 0" was the incorrect one, not the emulator). Investigate all envelope shapes that exercise the down+hold (alt=0) branch.

3. **Turbosound default panning and pan=01 routing** — audit flagged TS-10 ("Reset: all panning to both L+R") as a real emulator bug; Step 2 empirical run also shows TS-42 ("Pan 01: R only, L silenced") broken with `L=0 R=0`. Both point to the Turbosound/YM2149 mixer not initializing panning to `0b11` (both channels) at reset AND the pan register 01 decode being wrong. Likely a single missing init in the Turbosound class reset path plus a decode off-by-one.

4. **ULA frame completion timing** — `test/ula/ula_test.cpp` S13.14 "Frame complete after full T-states" shows `frame_done=0 after 69888 T-states`. The bug is that `frame_done` doesn't flip when the T-state counter reaches the frame boundary. Investigate the T-state accumulation path in the ULA/video-timing class.

   Original diagnosis: **MMU reset slot clobber** — `Mmu::reset()` sets `slots_[0]=0x00, slots_[1]=0x01` but VHDL `zxnext.vhd:4611-4612` explicitly assigns `MMU0 <= X"FF"; MMU1 <= X"FF"` on reset (0xFF = ROM sentinel). Failing tests: RST-01, RST-02 in `mmu_test.cpp`. This was originally pre-approved as a cheap Step 2 emulator fix but deferred per the "emulator bugs → Task 3" rule.

6. **DMA zero-length transfer** — `test/dma/dma_test.cpp` rows 9.8, 14.6, 14.7 (Wave 1 2026-04-15 refactor IDs — was 9.7/14.4 pre-refactor) show the DMA transfers 1 byte when `block_len==0`. Root cause in `src/peripheral/dma.cpp:560`: counter post-increment before the length check means the first byte always transfers regardless of `block_len`. VHDL `dma.vhd:426` compares `counter < block_len` before the transfer → FINISH_DMA with 0 bytes when both are 0. Fix: move the length check before the byte transfer.

6a. ~~**DMA byte-mode (R4 mode=00) transfers whole block instead of one byte**~~ — **REMOVED 2026-04-15 by Task 3 audit**. VHDL `dma.vhd:426` block-length check is NOT gated by `R4_mode_s`, so byte mode runs the full block like every other mode. The prior plan row oracle (`n==1`) was a misreading of the VHDL against the Z80-DMA external data-sheet semantics, not an emulator bug. Test row 12.6 now asserts `n==4` (full block) and passes against current C++. Not a bug.

6b. **DMA continuous+prescaler gap** — `test/dma/dma_test.cpp` row 12.7. `src/peripheral/dma.cpp:581` guards `burst_wait_` with `mode_ == 2` (burst only), so continuous mode (`mode_==1`) never enters prescaler wait. VHDL `dma.vhd:424` enters WAITING_CYCLES whenever the prescaler is nonzero, regardless of R4 mode. **NEW Task 2 backlog item from Wave 1 2026-04-15**. Fix: decouple the prescaler wait-arm from the mode check.

7. **NextREG reset-defaults tests rewrite as integration tests** — 14 RST-xx failures in `test/nextreg/nextreg_test.cpp` are a test-design issue, not an emulator bug. The bare `NextReg` class doesn't own the VHDL reset defaults (NR 0x12=0x08, 0x14=0xE3, 0x42=0x07, 0x4A=0xE3, 0x4B=0xE3, 0x50-0x57 MMU mirrors, 0x82-0x89 port enables). These defaults live in subsystem classes (Ula, Layer2, Mmu, port dispatcher) that hook handlers on NextReg. When the test constructs a bare `NextReg`, no subsystems are attached, so reads return zero. The real emulator is not broken — in a full `ZxNext` boot, port 0x253B reads return the correct VHDL values. **Fix direction**: rewrite these 14 tests as full-machine integration tests (same pattern as demo/regression tests): construct `ZxNext`, reset, read via port dispatcher, check against VHDL defaults. 9. **Ram class default size is 768 KB, should be 1792 KB** — `src/memory/ram.h:7` has `explicit Ram(size_t size_bytes = 768 * 1024)`. The `Emulator` class (`src/core/emulator.h:285`) holds `Ram ram_` with no explicit size, so the running emulator only has 768 KB of usable RAM instead of the ZX Next spec 1792 KB (224 × 8K pages). Any NEX program declaring `ram_required = 1` (1792 KB) silently fails to fit. Surfaced during Task 5 Step 4 while fixing the MMU test fixture (which was also at 768 KB — those test fixtures were bumped to 1792 KB in the L2M-01 rewrite commit). Verify: re-run all unit tests (they allocate explicitly, so no change expected) plus `bash test/regression.sh` and a NEX program that needs >768 KB to confirm the emulator actually uses the new space. Also audit `Ram::load_state` for any buffer-size mismatch when loading older snapshots.
