## Task 1: Continue parallax.nex investigation

Continuation of 2026-04-26 session. The visual delta between CSpect and
jnext on parallax.nex is **NOT explained by**: MMU mapping (byte-exact
verified), sprite anchor/relative, L2 image data, sprite multiplexing
across scanlines, CPU-speed timing skew, OR refresh-rate (50 vs 60 Hz —
user-verified at 50 Hz CSpect still shows rich content while jnext shows
sparse).

The remaining unexplained gap is a real jnext-side bug. Plan:

### Phase A (start here): sprite priority / transparency / pixel-level compositor

The 96 sprites at y=112/y=128 DO render in jnext, but maybe their
priority vs L2 / ULA is wrong, hiding content. Investigate:
- NR 0x4A (sprite transparency index) — current value `0xE3` from demo
- NR 0x4C (sprite/tilemap palette select) — current `0x0F` from demo
- NR 0x6F (sprite-over-border) — default
- NR 0x68 (ULA blend mode) — default
- NR 0x15 SLU/LSU priority bits — `0x01` from demo
- Compositor priority handling vs VHDL `zxnext.vhd`

Audit each NR's handler in jnext for VHDL faithfulness when written.
Add per-pixel tracing in the compositor for parallax frame 250 to
identify where layer choices differ from CSpect's expected output (we
don't have CSpect-side per-pixel data, but we can verify jnext is
internally consistent).

### Phase B: decode bank 6 main loop

Disassembly covered IM1 vector install at 0x0053 and per-frame loop at
0x0090, but per-frame work calls were NOT yet traced:
- CALL 0x011E
- CALL 0x01EF
- CALL 0x0213
- CALL 0x0269
- CALL 0x02BF
- CALL 0x00F8
- CALL 0x0469

Decode each: what NR / port writes the demo issues each frame.
Cross-reference against trace at `/tmp/parallax-investigation/full-trace/`.
Find any NR or port jnext mishandles when the demo writes it.

**Note**: Z80N `NEXTREG` instruction is `ED 91 nn vv` (4 bytes), but
z88dk-dis treats it as 2-byte NOP — manual realignment required after
each `ED 91` for accurate disassembly. The init code at PC=0x8000
(NEX bank 2 lo, file offset 16896) was successfully decoded by hand.

## Task 2: NR 0x07 turbo handling defect (open backlog)

Surfaced 2026-04-26: writing `NR 0x07 = 0x03` (turbo 28 MHz) in
`Emulator::nex_init_machine` causes `tilemap-demo` (regression test)
to render BLACK SCREEN. Confirmed interactively. Likely a jnext-side
defect in turbo handling — possibly:
- Tight scanline-synchronized demo loops calibrated for 3.5 MHz miss
  their target at 28 MHz (could be VHDL-faithful behaviour but
  unfortunate)
- Effective T-state accounting at 28 MHz has a bug in jnext

Investigation: build a minimal NEX demo that just writes `NR 0x07 = 0x03`
and observe what visibly fails vs CSpect. Compare jnext's NR 0x07
handler against VHDL `zxnext.vhd` reset/runtime values.
