## Task 1: Video panel bug fixes (continued from 2026-04-08)

Bugs fixed in VideoLayerView and debugger raster stepping:

- VideoLayerView image too small / clipped on fractional-DPR screens (DPR=1.25)
  - Root cause: `sizeHint()` used `primaryScreen()->devicePixelRatio()` (returned 2.0) but widget
    was placed on a 1.25× screen; fixed size was too small, image was clipped
  - Fix: removed `setFixedSize` from constructor; added `showEvent()` override that calls
    `setFixedSize(sizeHint())` after the widget knows its actual DPR
  - `sizeHint()` now uses `devicePixelRatioF()` (widget's own DPR) with primaryScreen fallback
  - `paintEvent` computes `phys_w/phys_h` from actual widget content area (not hardcoded 640×512)
  - `paintEvent` self-corrects size if stale (fixes first-open ULA tab appearing wrong)

- VideoPanel margins not symmetric
  - Added `setContentsMargins(12, 4, 12, 4)` to tab content layout for equal left/right padding
  - VideoPanel `sizeHint()` widened to 600×600

- VC counter showing 0..319 instead of 0..255
  - `LINES_PER_FRAME=320` but only 0..255 are visible (`Renderer::FB_HEIGHT=256`)
  - Clamped `paused_vc_` to `Renderer::FB_HEIGHT - 1` in `VideoPanel::refresh()`

- EOSL stepping cycling 320 times instead of 256
  - `on_run_to_eosl()`: when next scanline VC >= `FB_HEIGHT`, skip blanking and target
    `frame_start + MASTER_CYCLES_PER_FRAME` (next frame start) instead of next blanking line
  - This means stepping wraps 255→0 in 256 steps, skipping the 64 invisible blanking lines

- EOSL from VC=255 went to VC=1 instead of VC=0
  - Root cause: `snapshot_raster()` "between frames" detection (`elapsed < MASTER_CYCLES_PER_LINE`)
    was returning `last_frame_vc_=319` (clamped to 255) when at VC=0 of new frame.
    On the NEXT EOSL click, actual position was VC=0, so next-line was VC=1.
  - Fix: removed "between frames" detection from `snapshot_raster()`; always compute VC
    directly from `elapsed / MASTER_CYCLES_PER_LINE`. When between frames, elapsed≈0 → VC=0,
    which is accurate (CPU IS at the start of a new frame).

- EOF (run to end of frame) stopped at VC=0 showing black frame
  - After removing between-frames detection, `target = frame_start + MASTER_CYCLES_PER_FRAME`
    now correctly stops at VC=0 of next frame — but the video panel shows a black/unrendered frame
  - Fix: changed EOF target to midpoint of last visible scanline:
    `frame_start + (FB_HEIGHT-1)*MASTER_CYCLES_PER_LINE + MASTER_CYCLES_PER_LINE/2`
    so VC=255 is shown with full frame visible; if already past that, targets next frame

- Debugger window: left tab group now wider (`setSizes({450, 300})`, stretch 3:2)

- The Tilemap video panel does not follow the real emulator view. I believe the image constructed for the Tilemap layer does not take into account the HW scrolling registers, and so shows bumps when the screen is redrawn every once in a while with the updated tiles. The Tilemap view should take into account the scrolling registers and show the corrected picture.
  - Root cause: `VideoLayerView::refresh(vc)` had `if (vc == last_vc_) return;` which skips
    re-rendering when vc is unchanged. When stepping EOF→EOF, the emulator always stops at
    vc=255, so the panel never re-rendered — displaying stale scroll/tile state from the
    previous frame. The scrolling code itself was correct; it just wasn't being invoked.
  - Fix: only skip re-render when showing the "running dim placeholder" (`vc < 0 && last_vc_ < 0`).
    When paused (vc ≥ 0), always re-render since registers may have changed between frames.
