## Task 2: investigate speed bug

- When I run at normal speed (3.5 MHz) normal spectrum games seem to go quite faster than expected. There seems to be some issue calculating the timing in the emulator loop?
- Not working. Still goes faster. It even reports running at 53 FPS, but very fast I believe it's even faster
- The FPS is 50 now in the GUI, but the game still show quite fast, I think it's about 50% faster than normal. Deep analysis please.

### Findings

**Root cause:** `MASTER_CYCLES_PER_FRAME` is hardcoded to 560,000 (28 MHz / 50 Hz exactly) and `LINES_PER_FRAME` to 320, but the VHDL shows these values are machine-type-dependent and none are exactly 50 Hz:

| Machine  | Pixels/line | Lines/frame | Master cycles/line | Master cycles/frame | True frame rate |
|----------|-------------|-------------|--------------------|---------------------|-----------------|
| 48K      | 448         | 312         | 1792               | 558,144             | 50.17 Hz        |
| 128K     | 456         | 311         | 1824               | 567,264             | 49.36 Hz        |
| Pentagon | 448         | 320         | 1792               | 573,440             | 48.83 Hz        |
| **Ours** | **N/A**     | **320**     | **1750**           | **560,000**         | **50.00 Hz**    |

**Effect on 48K mode:** We execute 560,000 master cycles per frame but 48K hardware does 558,144. At 3.5 MHz (divisor 8), that's 70,000 vs 69,768 T-states — **~0.3% extra per frame**. However, the real issue is that `TSTATES_PER_LINE = 228` (128K value) is used for 48K mode where it should be 224. This means `228 × 8 = 1824` master cycles per interrupt offset instead of `224 × 8 = 1792`, so the **ULA interrupt fires 32 master cycles late** in 48K mode.

**But the bigger issue is:** The emulator paces at 20ms/frame (50.00 Hz) via Qt timer, while real 48K hardware runs at 50.17 Hz. This alone would make games run ~0.3% slower, not faster. The "games feel fast" perception might come from:
1. No contention delays being applied in some cases (making CPU faster than real hardware)
2. The interrupt timing being wrong (firing at wrong position, affecting frame-locked game loops)

**Fix needed:** Make `MASTER_CYCLES_PER_FRAME`, `MASTER_CYCLES_PER_LINE`, and `LINES_PER_FRAME` machine-type-dependent instead of global constants.
